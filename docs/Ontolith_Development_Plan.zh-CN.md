## 1. 目标

本计划定义 Ontolith 从 R1 到 R4 的完整实施路径，并与架构规范、Rust-only 生产策略、第三方组件受控复用策略保持一致。

本文档包含两种视图：
- 评审视图：范围、决策、约束、风险。
- 执行视图：WBS、依赖关系、里程碑、验收标准、交付节奏。

## 3. 评审视图

### 3.1 范围

包含：
- R1 到 R4 的端到端技术交付。
- 运行时架构实现与工程化流程。
- 安全、可观测性、发布流程与依赖治理。
- 对 Oxigraph、RocksDB 及 Rust 核心生态依赖的受控管理。

不包含：
- 商业定价与市场策略。
- 非技术组织绩效体系。
- 独立 UI 产品设计细节。

### 3.2 核心决策

- 控制面和数据面生产路径全部采用 Rust。
- 第三方原生引擎仅允许通过 Rust 适配层接入。
- Oxigraph 作为语义内核复用候选，不作为平台边界。
- RocksDB 通过存储抽象层接入，禁止上层泄露厂商 API。
- RDF/SPARQL 标准符合性纳入发布硬门禁。

### 3.3 架构约束

- 存储、查询、插件边界必须通过内部抽象层暴露。
- 对不支持标准特性的错误行为必须确定且可文档化。
- 一致性级别与事务语义必须在 API 层明确声明。
- 新增 Tier A 依赖或架构例外必须走 RFC/ADR 审批。

### 3.4 主要风险与缓解

- 范围膨胀风险：使用 R1 非目标与特性开关强约束。
- 分布式复杂度风险：先单区域稳定，再推进高级集群能力。
- 推理性能风险：推理模式可配置，避免热路径深推理。
- 依赖绑定风险：接口隔离 + 兼容矩阵 + 可替换策略。
- 团队爬坡风险：Rust 规范、模板与分阶段培训并行。

## 4. 执行视图

### 4.1 交付阶段

目标：

交付物：
- 已批准范围基线。
- 架构例外审批模板。
- 依赖登记模板与评审规则。

目标：
- 实现 Knowledge Object 领域模型。
- 实现 Node 标识与字典管理器。
- 定义存储抽象接口。

交付物：
- ontolith-core 模型基线。
- 双向映射字典服务。
- 稳定的存储 trait 契约。

依赖：
- 阻塞 Phase 2 与 Phase 3。

## 5. 工作分解结构（WBS）

### WBS-01 核心运行时与知识模型
- Resource、Statement、Graph、Dataset、Ontology 结构。
- 确定性标识与规范化编码规则。

### WBS-02 解析与导入
- MVP 范围 RDF 语法支持。
- 流式解析链路与错误契约。

### WBS-03 存储与事务
- 字典管理器。
- 三元组/四元组物理编码。
- WAL、MVCC、快照与恢复。

### WBS-04 查询与优化
- SPARQL 解析与代数转换。
- 逻辑/物理计划与迭代执行器。
- Explain、超时、取消。

### WBS-05 推理与 SHACL
- OWL 2 RL 核心规则。
- SHACL 基线约束与报告。

### WBS-06 分布式运行时
- 元数据服务与共识。
- 分片、复制、故障转移与重平衡。

### WBS-07 API、安全与集成
- 网关边界。
- 鉴权、租户隔离、审计链路。
- 插件宿主与扩展生命周期。

### WBS-08 平台工程
- CI/CD、质量门禁、依赖审计、发布流程。
- 可观测性栈与运维手册。

## 6. 里程碑（R1 到 R4）

### R1 MVP（Rust）
范围：
- RDF 核心运行时。
- SPARQL 查询基线。
- 单区域集群核心。
- 安全与审计基线。

退出标准：
- MVP 范围标准符合性门禁通过。
- 核心 SLO 基线达标。
- 恢复演练与回滚演练均通过。

### R2（Rust）
范围：
- 代价优化。
- OWL 2 RL 核心。
- SHACL 基线。

退出标准：
- Explain 质量与优化稳定性门禁通过。
- 推理正确性与性能护栏门禁通过。

### R3（Rust）
范围：
- 高级集群运维能力。
- GeoSPARQL 范围能力。
- 企业级安全加固。

退出标准：
- 高可用与故障转移门禁通过。
- 租户隔离与审计加固门禁通过。

### R4（Rust）
范围：
- AI-native 语义运行时扩展。

退出标准：
- 扩展安全与兼容门禁通过。
- 检索与语义集成 KPI 达标。

## 7. 验收与质量门禁

### 7.1 功能正确性
- 支持范围内 RDF/SPARQL 标准测试通过。
- API 契约与错误语义可重复、可预测。

### 7.2 一致性与可靠性
- 故障注入验证选主、复制、恢复行为。
- 重试场景下幂等写入行为可验证。

### 7.3 性能与容量
- 延迟、吞吐、RTO、RPO 满足发布目标。
- 性能回归超过阈值时阻断候选发布。

### 7.4 安全与合规
- 鉴权策略与租户隔离测试通过。
- 许可证审计与漏洞审计通过。

### 7.5 工程质量
- cargo fmt --check、cargo clippy -D warnings 与全量测试通过。
- 安全敏感模块建议执行 Miri/sanitizer 检查。

## 8. 第三方组件治理计划

### 8.1 分级
- Tier A：运行关键依赖。
- Tier B：可选运行依赖，需降级路径。
- Tier C：构建、测试、开发依赖。

### 8.2 初始基线
- Oxigraph：语义内核复用候选。
- RocksDB（Rust 绑定）：耐久与存储候选。
- Tokio、Serde、Tracing 生态：运行与平台基线。

### 8.3 准入规则
- 新增 Tier A 依赖必须通过 RFC 与 ADR。
- 每个依赖必须定义 owner、风险级别、回退与替换方案。
- 禁止厂商特定 API 上浮到内部抽象层之外。

### 8.4 版本与供应链策略
- 生产依赖禁止通配版本。
- 提交 Cargo.lock 以支持可复现构建。
- CI 持续执行漏洞与依赖审计。
- 严重漏洞修复目标时限：72 小时。

## 9. 团队与运行模式

### 9.1 建议并行流
- Stream A：核心存储与事务。
- Stream B：查询与推理。
- Stream C：分布式运行时。
- Stream D：平台安全与运维。

### 9.2 节奏建议
- R1 目标：2-3 个月形成可验收 MVP。
- R2 与 R3：按季度迭代并执行硬门禁。
- 每阶段预留 15%-20% 容量用于稳定性与回归修复。
